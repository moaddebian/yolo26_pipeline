Les Étapes d'Entraînement de YOLO26 — Rien n'est Sauté

Chaque étape qui se déroule depuis le chargement des données jusqu'au calcul du gradient, pour UNE itération d'entraînement

0

Initialisation du modèle et des losses

Avant toute chose, le modèle et le système de perte sont créés :

# Le modèle YOLO26 est créé avec end2end=True # head.py — Detect.__init__() nc = 80 # nombre de classes (COCO) reg_max = 1 # PAS de DFL ! Régression directe des coordonnées end2end = True # Active la double tête # Tête O2M (one-to-many) — cv2 pour box, cv3 pour classification self.cv2 = [Conv→Conv→Conv2d(out=4)] × 3 niveaux # régression de boîtes self.cv3 = [DWConv→Conv→DWConv→Conv→Conv2d(out=80)] × 3 # classification # Tête O2O (one-to-one) — copie INDÉPENDANTE de la tête O2M self.one2one_cv2 = copy.deepcopy(self.cv2) # mêmes couches, poids DIFFÉRENTS self.one2one_cv3 = copy.deepcopy(self.cv3) self.dfl = nn.Identity() # reg_max=1 → pas de DFL
# loss.py — E2ELoss.__init__() # Deux fonctions de perte avec des TAL DIFFÉRENTS self.one2many = v8DetectionLoss(model, tal_topk=10) self.one2one = v8DetectionLoss(model, tal_topk=7, tal_topk2=1) # ProgLoss — poids initiaux self.o2m = 0.8 # début : 80% pour O2M self.o2o = 0.2 # début : 20% pour O2O self.final_o2m = 0.1 # fin : 10% pour O2M
⚡ Pourquoi ?

deepcopy : Les deux têtes doivent avoir la même architecture mais des poids indépendants. O2O apprend à ne produire qu'UNE prédiction par objet ; O2M en produit 10. Si elles partageaient les mêmes poids, elles ne pourraient pas avoir des comportements différents.

reg_max=1 : Supprime la DFL. Au lieu de prédire une distribution softmax sur 16 valeurs, le réseau prédit directement 4 distances (l, t, r, b). Simplifie l'export NPU/DSP.

▼
1

Chargement d'un batch de données

Le dataloader charge un batch d'images avec leurs annotations standard COCO :

# Données standard — AUCUNE étiquette spéciale pour NMS-Free batch = { "img": tensor[B, 3, 640, 640], # B images RGB redimensionnées "batch_idx": tensor[N], # index image de chaque annotation "cls": tensor[N, 1], # classe de chaque objet (0-79) "bboxes": tensor[N, 4], # boîtes normalisées [x, y, w, h] } # Exemple : 16 images, 127 objets annotés au total
⚡ Pourquoi ?

Le NMS-Free ne nécessite AUCUNE modification du dataset. Ce sont des annotations COCO standard (une boîte + une classe par objet). Le "secret" du NMS-Free est dans le label assignment, PAS dans les données.

▼
2

Forward pass — Backbone + Neck (partagés)

L'image traverse le backbone et le neck, produisant des feature maps multi-échelle :

# Backbone : extraire des features visuelles à différentes résolutions # Neck (FPN/PAN) : fusionner les features multi-échelle # Sortie : 3 niveaux de features features = [ P3: tensor[B, C, 80, 80], # stride=8 → 6400 ancres (détails fins) P4: tensor[B, C, 40, 40], # stride=16 → 1600 ancres P5: tensor[B, C, 20, 20], # stride=32 → 400 ancres (vue globale) ] # Total : 8400 ancres par image
⚡ Pourquoi 3 niveaux ?

P3 (80×80) détecte les petits objets (piétons éloignés). P4 (40×40) les objets moyens. P5 (20×20) les grands objets. Cette hiérarchie est essentielle pour détecter des objets de toutes tailles.

▼
3

Forward pass — Tête O2M (one-to-many)

Les features passent par la tête O2M avec les features normales (gradient activé) :

# head.py — Detect.forward() preds_o2m = self.forward_head(x, box_head=self.cv2, cls_head=self.cv3) # forward_head() détaillé : for i in range(3): # pour chaque niveau P3, P4, P5 boxes_i = self.cv2[i](x[i]) # Conv→Conv→Conv2d → prédictions de boîtes scores_i = self.cv3[i](x[i]) # DWConv→Conv→...→Conv2d → scores de classes # Résultat O2M : preds_o2m = { "boxes": tensor[B, 4, 8400], # 4 coordonnées × 8400 ancres "scores": tensor[B, 80, 8400], # 80 scores de classe × 8400 ancres "feats": [P3, P4, P5], # features originales }
⚡ Pourquoi ?

La tête O2M utilise les features avec gradient. Ses gradients remontent au backbone et au neck. C'est elle qui entraîne principalement le backbone car elle produit un signal dense (10 ancres positives par objet).

▼
4

Forward pass — DETACH + Tête O2O (one-to-one)

Les features sont détachées puis passées à la tête O2O :

# head.py — Detect.forward() — LIGNE CRUCIALE x_detach = [xi.detach() for xi in x] # ← COUPE LE GRADIENT ! preds_o2o = self.forward_head(x_detach, box_head=self.one2one_cv2, cls_head=self.one2one_cv3 ) # Résultat O2O : preds_o2o = { "boxes": tensor[B, 4, 8400], # mêmes dimensions que O2M "scores": tensor[B, 80, 8400], # mais poids DIFFÉRENTS "feats": [P3_detach, P4_detach, P5_detach], }
⚡ Pourquoi detach() ?

C'est la ligne la plus importante pour la stabilité de l'entraînement. Sans detach(), la tête O2O enverrait des gradients au backbone qui CONTREDISENT ceux de O2M. O2M veut activer 10 ancres par objet, O2O veut n'en activer qu'une seule. Ces signaux opposés rendraient l'entraînement du backbone instable et divergent.

Avec detach(), seuls les poids de one2one_cv2 et one2one_cv3 sont mis à jour par la loss O2O. Le backbone est entraîné UNIQUEMENT par O2M.

▼
5

Pré-traitement des targets

Les annotations COCO sont converties au format interne :

# loss.py — v8DetectionLoss.get_assigned_targets_and_loss() # 1. Concaténer batch_idx + cls + bboxes targets = torch.cat((batch["batch_idx"], batch["cls"], batch["bboxes"]), 1) # 2. Convertir xywh → xyxy et mettre à l'échelle de l'image targets = self.preprocess(targets, batch_size, scale=imgsz) # → tensor[B, max_obj, 5] (cls + x1 + y1 + x2 + y2) # 3. Séparer labels et boîtes gt_labels, gt_bboxes = targets.split((1, 4), dim=2) mask_gt = gt_bboxes.sum(2).gt_(0.0) # masque des GT valides # 4. Générer les points d'ancrage anchor_points, stride_tensor = make_anchors(feats, stride, 0.5) # anchor_points: tensor[8400, 2] — coordonnée (x,y) de chaque ancre # stride_tensor: tensor[8400, 1] — stride de chaque ancre (8, 16, ou 32) # 5. Décoder les boîtes prédites pred_bboxes = self.bbox_decode(anchor_points, pred_distri) # YOLO26 (reg_max=1) : dist2bbox(pred_dist, anchors) directement # PAS de softmax, PAS de DFL — juste anchor ± distance
⚡ Pourquoi make_anchors() ?

Les ancres sont des points sur une grille régulière. Pour une image 640×640 : P3 produit 80×80 = 6400 points espacés de 8px, P4 → 1600 points de 16px, P5 → 400 points de 32px. Total : 8400 points. Chaque point est un candidat potentiel pour prédire un objet.

▼
6

Label Assignment — TAL (le cœur du NMS-Free)

C'est l'étape la plus critique. Le TAL décide QUELLES ancres sont positives pour QUEL objet. Voir l'onglet "TAL en Détail" pour l'explication complète.

# loss.py — Appel du TAL — FAIT SÉPARÉMENT pour O2M et O2O # Pour la tête O2M : assigner_o2m = TaskAlignedAssigner(topk=10, alpha=0.5, beta=6.0) # → Assigne jusqu'à 10 ancres par objet # Pour la tête O2O : assigner_o2o = TaskAlignedAssigner(topk=7, topk2=1, alpha=0.5, beta=6.0) # → Pré-sélectionne 7 candidates, puis ne garde que LA MEILLEURE # Résultat de chaque assignation : _, target_bboxes, target_scores, fg_mask, _ = assigner( pred_scores.detach().sigmoid(), # scores détachés (pas de gradient ici) pred_bboxes.detach(), # boîtes détachées anchor_points, gt_labels, gt_bboxes, mask_gt ) # fg_mask: tensor[B, 8400] — True pour les ancres positives # target_scores: tensor[B, 8400, 80] — scores cibles CONTINUS (pas binaires !)
⚡ Pourquoi topk=7 puis topk2=1 ?

Si on mettait directement topk=1, le TAL choisirait la meilleure ancre sur la base d'une seule mesure. Avec topk=7, on explore d'abord 7 candidates par objet, on calcule leurs métriques d'alignement, on résout les conflits (une ancre ne peut servir qu'un seul objet), PUIS on filtre à topk2=1. Cela donne une sélection plus robuste et stable.

▼
7

Calcul des 3 pertes (pour chaque tête)

Chaque tête (O2M et O2O) calcule indépendamment 3 composantes de perte :

# loss.py — v8DetectionLoss.get_assigned_targets_and_loss() loss = torch.zeros(3) # [box, cls, dfl] ## ① PERTE DE CLASSIFICATION (BCE) loss[1] = BCE(pred_scores, target_scores) / target_scores_sum # Compare les scores prédits aux scores cibles CONTINUS # target_scores ne sont PAS des {0,1} mais des valeurs dans [0,1] # proportionnelles à la qualité de l'alignement (IoU * score) ## ② PERTE DE LOCALISATION (CIoU) weight = target_scores.sum(-1)[fg_mask] # poids = qualité du target iou = bbox_iou(pred_bboxes[fg_mask], target_bboxes[fg_mask], CIoU=True) loss[0] = ((1.0 - iou) * weight).sum() / target_scores_sum ## ③ PERTE DE DISTANCE L1 (remplace DFL dans YOLO26) # YOLO26 (reg_max=1) : pas de DFL → perte L1 normalisée target_ltrb = bbox2dist(anchor_points, target_bboxes) # Normaliser par la taille de l'image target_ltrb = target_ltrb * stride / imgsz pred_dist = pred_dist * stride / imgsz loss[2] = L1_loss(pred_dist[fg_mask], target_ltrb[fg_mask]) * weight # Appliquer les gains des hyperparamètres loss[0] *= hyp.box # gain box (ex: 7.5) loss[1] *= hyp.cls # gain cls (ex: 0.5) loss[2] *= hyp.dfl # gain dfl (ex: 1.5)
⚡ Pourquoi CIoU et pas L1 pour les boîtes ?

CIoU mesure la qualité géométrique globale (chevauchement + distance centres + ratio d'aspect). L1 ne mesure que les erreurs coordonnée par coordonnée sans notion de forme. Les deux sont complémentaires : CIoU pour la qualité spatiale, L1 pour la précision des distances aux bords.

Pourquoi les targets sont continus et pas binaires ? Une ancre qui chevauche 90% du GT mérite un target de ~0.9, tandis qu'une ancre qui chevauche 30% mérite ~0.3. Cela guide la BCE à pénaliser proportionnellement.

▼
8

ProgLoss — Pondération progressive

Les pertes des deux têtes sont combinées avec des poids qui changent à chaque époque :

# loss.py — E2ELoss.__call__() # Calcul des losses des deux têtes loss_o2m = self.one2many.loss(preds["one2many"], batch) loss_o2o = self.one2one.loss(preds["one2one"], batch) # ProgLoss : pondération progressive L_total = loss_o2m * self.o2m + loss_o2o * self.o2o # À chaque fin d'époque : # loss.py — E2ELoss.update() self.updates += 1 self.o2m = self.decay(self.updates) self.o2o = max(1.0 - self.o2m, 0) # Formule de décroissance linéaire : def decay(self, x): return max(1 - x / (epochs - 1), 0) * (0.8 - 0.1) + 0.1

⚡ Pourquoi une progression linéaire ?

Début (w_o2m=0.8) : Le modèle ne sait rien. Il a besoin du signal dense de O2M (10 ancres/objet) pour apprendre les features de base du backbone rapidement.

Milieu : Le backbone est entraîné. On commence à renforcer O2O pour que la tête O2O affine sa sélection unique.

Fin (w_o2m=0.1) : On veut que le modèle s'aligne au maximum avec son comportement à l'inférence (O2O seule). Le poids O2M reste à 0.1 (pas 0) pour maintenir une régularisation légère.

9

Backpropagation et mise à jour des poids

Le gradient est calculé et les poids sont mis à jour par MuSGD :

# Backpropagation L_total.backward() # Les gradients circulent DIFFÉREMMENT selon la tête : # # L_o2m → cv2, cv3 → Neck → Backbone ← GRADIENT COMPLET # L_o2o → one2one_cv2, one2one_cv3 → STOP ← GRADIENT BLOQUÉ par detach() # # Résultat : le backbone est entraîné UNIQUEMENT par O2M # Les têtes O2O sont entraînées indépendamment # MuSGD : mise à jour hybride optimizer.step() # Backbone → SGD + correction Muon (stabilisation) # Têtes → SGD standard (apprentissage rapide)
⚡ Pourquoi MuSGD ?

L'entraînement dual-head est plus instable qu'un entraînement classique. MuSGD ajoute une "correction de courbure" au backbone, inspirée de l'optimisation des LLM (Muon/Kimi K2), qui lisse les oscillations de gradient et accélère la convergence.

10

Inférence — Tête O2O uniquement

À l'inférence, la tête O2M est complètement supprimée. Seule O2O est utilisée :

# head.py — Detect.forward() en mode inférence # self.training == False # 1. N'utilise QUE la tête O2O y = self._inference(preds["one2one"]) # → decode boxes + sigmoid(scores) # 2. Post-process : top-300 par score (PAS de NMS !) y = self.postprocess(y) # → tensor[B, 300, 6] # Format : [x1, y1, x2, y2, score, class_id] # postprocess() détaillé : boxes, scores = preds.split([4, 80], dim=-1) scores, conf, idx = self.get_topk_index(scores, max_det=300) # → Sélectionne les 300 ancres avec les scores les plus élevés # PAS de NMS, PAS de suppression par IoU, juste un tri par score
⚡ Pourquoi ça marche sans NMS ?

Parce que pendant l'entraînement, la tête O2O a appris (grâce au TAL topk2=1) à n'activer qu'UNE SEULE ancre par objet. Les 8399 autres ancres ont été entraînées à prédire un score FAIBLE. Donc en sélectionnant les top-300, on obtient naturellement ~1 prédiction par objet sans aucun doublon.

La méthode fuse() supprime même les couches O2M du modèle : self.cv2 = self.cv3 = None

Le TaskAlignedAssigner (TAL) — Étape par Étape

L'algorithme qui décide quelle ancre est responsable de quel objet — le cœur du NMS-Free

A

Entrée : prédictions + ground truth

# tal.py — TaskAlignedAssigner.forward() # Appelé avec @torch.no_grad() — pas de gradient dans le TAL ! pd_scores # tensor[B, 8400, 80] — scores de classification (sigmoid) pd_bboxes # tensor[B, 8400, 4] — boîtes prédites (xyxy, en pixels) anc_points # tensor[8400, 2] — centres des ancres (x, y) gt_labels # tensor[B, M, 1] — classes ground truth gt_bboxes # tensor[B, M, 4] — boîtes ground truth (xyxy) mask_gt # tensor[B, M, 1] — masque des GT valides
▼
B

Étape 1 — Filtrage géométrique + STAL

On garde uniquement les ancres dont le centre est à l'intérieur d'une boîte GT :

# tal.py — select_candidates_in_gts() # ① STAL : élargir les petites GT gt_xywh = xyxy2xywh(gt_bboxes) wh_mask = gt_xywh[..., 2:] < stride[0] # objets plus petits que 8px gt_xywh[..., 2:] = where(wh_mask, 16, gt_xywh[..., 2:]) # → Les objets <8px sont élargis à 16px pour le calcul de candidats # → Garantit des ancres candidates même pour les tout petits objets gt_bboxes = xywh2xyxy(gt_xywh) # ② Test d'inclusion : le centre de l'ancre est-il dans la GT ? lt, rb = gt_bboxes.chunk(2) # left-top, right-bottom deltas = cat(anc_points - lt, rb - anc_points) mask_in_gts = deltas.amin(-1).gt_(0) # → tensor[B, M, 8400] — True si l'ancre est dans la GT # Résultat : sur 8400 ancres, peut-être 20-50 sont dans la GT box
⚡ Pourquoi STAL ?

Un piéton à 100m vu depuis un drone occupe peut-être 5×10 pixels. Avec stride=8, il n'y a AUCUN centre d'ancre dans une boîte si petite. STAL élargit artificiellement ces boîtes à 16px pour la sélection des candidats (pas pour le calcul de la perte). Sans STAL, ces petits objets n'auraient zéro ancre positive et seraient invisibles à l'entraînement.

▼
C

Étape 2 — Calcul de la métrique d'alignement

Pour chaque paire (ancre candidate, GT), on calcule un score de qualité :

# tal.py — get_box_metrics() # ① Extraire le score de classification pour la classe du GT bbox_scores[mask] = pd_scores[batch_idx, :, gt_class][mask] # → Pour chaque ancre, prendre le score de la classe correspondante # ② Calculer l'IoU entre la boîte prédite et la GT overlaps[mask] = bbox_iou(gt_boxes, pd_boxes, CIoU=True) # ③ Métrique d'alignement = classification × localisation align_metric = bbox_scores^0.5 * overlaps^6.0 # alpha=0.5 (classification) << beta=6.0 (localisation) # → La LOCALISATION est 12× plus importante que la classification !
⚡ Pourquoi beta=6 >> alpha=0.5 ?

Pour un détecteur NMS-Free, il est crucial que l'ancre sélectionnée soit la mieux localisée, pas simplement la plus confiante. Pourquoi ? Parce qu'il n'y aura PAS de NMS pour corriger une mauvaise sélection après coup. Si l'ancre choisie a un bon score de classification mais une mauvaise boîte, la détection finale sera mal positionnée et il n'y a aucun mécanisme de rattrapage.

Exemple : ancre A a score_cls=0.9 et IoU=0.3 → M = 0.9^0.5 × 0.3^6 = 0.95 × 0.0007 = 0.0007. Ancre B a score_cls=0.5 et IoU=0.8 → M = 0.5^0.5 × 0.8^6 = 0.71 × 0.262 = 0.186. L'ancre B (mieux localisée) gagne largement.

▼
D

Étape 3 — Sélection top-k

On garde les k meilleures ancres par objet (k dépend de O2M ou O2O) :

# tal.py — select_topk_candidates() # Sélection des top-k par métrique d'alignement topk_metrics, topk_idxs = torch.topk(align_metric, self.topk, dim=-1) # Créer un masque binaire des top-k count_tensor = torch.zeros(metrics.shape) for k in range(self.topk): count_tensor.scatter_add_(-1, topk_idxs[:,:,k:k+1], 1) count_tensor.masked_fill_(count_tensor > 1, 0) # Masque final = topk ∩ dans_GT ∩ GT_valide mask_pos = mask_topk * mask_in_gts * mask_gt # Pour O2M (topk=10) : ~10 ancres positives par objet # Pour O2O (topk=7) : ~7 ancres candidates par objet (pour l'instant)
▼
E

Étape 4 — Résolution des conflits + Filtrage topk2

Si une ancre est assignée à 2 objets différents, on la donne à celui avec le meilleur IoU. Puis pour O2O, on filtre à topk2=1 :

# tal.py — select_highest_overlaps() # ① Détecter les ancres assignées à plusieurs GT fg_mask = mask_pos.sum(dim="gt_axis") # (B, 8400) if fg_mask.max() > 1: # conflit détecté ! # Donner l'ancre au GT avec le meilleur IoU max_overlaps_idx = overlaps.argmax(dim="gt_axis") mask_pos = where(conflit, garder_meilleur_IoU, mask_pos) # ② FILTRAGE topk2 — UNIQUEMENT pour O2O (topk2 ≠ topk) if self.topk2 != self.topk: # O2O: topk=7, topk2=1 → True align_metric = align_metric * mask_pos # Garder seulement le TOP-1 par GT best_idx = torch.topk(align_metric, self.topk2, dim=-1).indices topk_idx = torch.zeros(mask_pos.shape) topk_idx.scatter_(-1, best_idx, 1.0) mask_pos *= topk_idx # ← NE GARDE QU'UNE SEULE ANCRE PAR OBJET # Résultat final : # O2M : ~10 ancres positives par objet → signal dense # O2O : EXACTEMENT 1 ancre positive par objet → NMS-Free
⚡ C'est ICI que le NMS-Free se produit !

Ce bloc if self.topk2 != self.topk est LE mécanisme qui rend YOLO26 NMS-Free. En forçant exactement 1 ancre positive par objet pendant l'entraînement, le réseau apprend que chaque objet ne doit activer qu'UN SEUL point de la grille. À l'inférence, il reproduit ce comportement naturellement.

▼
F

Étape 5 — Normalisation des scores cibles

Les scores cibles sont normalisés pour refléter la qualité de l'assignation :

# tal.py — _forward() — Normalisation finale align_metric *= mask_pos pos_align = align_metric.amax(dim=-1) # meilleur score par GT pos_overlaps = (overlaps * mask_pos).amax(dim=-1) # meilleur IoU par GT # Normaliser les métriques norm = (align_metric * pos_overlaps / (pos_align + eps)).amax(dim="gt_axis") # Appliquer au one-hot score target_scores = target_scores * norm # target_scores ∈ [0, 1] — pas binaire ! # Une ancre bien alignée (IoU élevé) → target_score ≈ 0.9 # Une ancre marginale → target_score ≈ 0.3
⚡ Pourquoi des targets continus ?

Si les targets étaient binaires {0, 1}, la BCE traiterait toutes les ancres positives de la même façon. Avec des targets continus, le réseau apprend à prédire des scores proportionnels à la qualité réelle de chaque ancre. Cela est essentiel pour O2O : l'unique ancre sélectionnée doit avoir un score de confiance qui reflète sa véritable qualité de localisation.

Résumé Technique — Répondre à "Pourquoi ?"

Chaque composant justifié, chaque décision expliquée

La question fondamentale

Comment faire pour qu'un réseau de neurones produise exactement 1 prédiction par objet, sans post-traitement ?

La réponse tient en une phrase : on l'entraîne à le faire. Pendant l'entraînement, on ne récompense qu'UNE SEULE ancre par objet (via le TAL avec topk2=1). Le réseau apprend que toutes les autres ancres doivent prédire un score faible (background). Ainsi, à l'inférence, il ne produit naturellement qu'une seule prédiction forte par objet.

Tête O2M Tête auxiliaire avec 10 ancres/objet (entraînement seul) Signal dense pour entraîner le backbone. O2O seul = signal trop faible -2 à -3% mAP. Backbone sous-entraîné, petits objets ratés

TAL topk2=1 Ne garde que la meilleure ancre par objet pour O2O Force le réseau à n'activer qu'1 point par objet Le réseau active plusieurs ancres → doublons → NMS nécessaire
ProgLoss Poids O2M : 0.8→0.1 ; O2O : 0.2→0.9 Début : apprendre les features (O2M). Fin : s'aligner sur l'inférence (O2O) Compromis sous-optimal. Le modèle ne s'aligne pas bien avec l'inférence
STAL Élargit les GT < 8px à 16px pour la sélection d'ancres Garantit des ancres candidates pour les très petits objets Piétons éloignés = 0 ancre positive → invisibles à l'entraînement
Sans DFL reg_max=1 → régression directe (pas de softmax) Simplifie l'export NPU/DSP. Pas de softmax → compatible partout Modèle fonctionne mais impossible à exporter sur certains edge devices
MuSGD SGD + correction Muon pour le backbone Stabilise l'entraînement dual-head qui est naturellement plus instable Convergence plus lente, instabilité possible sur les gros modèles
beta=6.0 La localisation pèse 12× plus que la classification dans TAL Pour O2O, l'ancre choisie DOIT être la mieux localisée (pas de NMS pour corriger) Ancre confiante mais mal placée sélectionnée → détections décalées

La chaîne de causalité complète

Voici le raisonnement "pourquoi ?" de bout en bout :

PROBLÈME : NMS est lent, non-parallélisable, et difficile à exporter ↓ pourquoi ne pas simplement le retirer ? PROBLÈME : Sans NMS, O2M produit ~10 doublons par objet ↓ pourquoi pas entraîner avec 1 ancre/objet directement ? PROBLÈME : Avec topk=1, signal trop faible → backbone sous-entraîné ↓ comment avoir un signal dense ET une sortie unique ? SOLUTION : Double tête O2M (dense) + O2O (unique) ↓ mais les gradients se contredisent... SOLUTION : detach() les features pour O2O ↓ mais O2M domine au début, O2O n'apprend pas assez... SOLUTION : ProgLoss : augmenter O2O progressivement ↓ mais comment choisir LA meilleure ancre pour O2O ? SOLUTION : TAL avec topk=7→topk2=1 + beta=6 (favoriser localisation) ↓ mais les petits objets n'ont pas d'ancres candidates... SOLUTION : STAL élargit les petites GT pour la sélection ↓ et pour le déploiement edge ? SOLUTION : Supprimer DFL (pas de softmax) + MuSGD pour stabilité ↓ RÉSULTAT : YOLO26 NMS-Free ✓

Ce qu'il faut pour implémenter NMS-Free (résumé pour votre encadrant)

Pour rendre N'IMPORTE QUEL détecteur NMS-Free, il faut :

  • Un label assigner one-to-one pendant l'entraînement (TAL topk2=1)
  • Un signal auxiliaire dense pour compenser le signal faible de O2O (tête O2M + signal dense topk=10)
  • L'isolation des gradients entre les deux objectifs (detach)
  • Un ordonnancement progressif des poids (ProgLoss)
  • Un mécanisme pour les petits objets (STAL ou équivalent)
  • Un assignation qui privilégie la localisation (beta >> alpha)

Sans n'importe lequel de ces éléments, soit le NMS-Free ne fonctionne pas du tout, soit les performances se dégradent significativement.
